# ワークスペース機能 仕様書

単一データ保持からの脱却。シートをディレクトリ形式で管理し、シート間で行データを
やり取りできるようにする。ベータ脱出の中核となる変更。

対象ブランチ: `feature/workspace-tree`

実装状況: Phase 1・2 完了 (§6)。Phase 3 以降は未着手。

---

## 1. 背景

現状のデータモデルは「1計画 = アプリ全体」になっている。

- `useStore.current_data: SaveData` が唯一の実体
- localStorage のキーは `'cost-manager'` 固定
- 別の計画を作るには既存を上書きするか、共有URLを発行して退避するしかない
- 共有URLの発行は TinyURL への外部通信に依存しており、失敗時のフォールバックがない

「艦これのシミュ結果をスプレッドシートのように管理する」というコンセプトに対して、
データ管理の側面が未達の状態にある。

### 決定事項

| 項目 | 決定 |
|---|---|
| シートの粒度 | ユーザー裁量。イベント全体を1シートに載せられること |
| フォルダのネスト | 無制限 |
| 行のコピペ | 行単位。複数行選択に対応する |
| Undo のスコープ | シート内の編集のみ。ツリー操作は対象外 |

## 2. データモデル

### 2.1 ツリー(正規化フラット)

ネスト無制限のため、`children` の再帰配列ではなく `parent_id` を持つフラットな
レコード集合として保持する。移動が1レコードの更新で済み、循環チェックが親を辿る
だけで書ける。

```ts
type NodeId = Brand<string, 'NodeId'>;

type BaseNode = {
    id: NodeId,
    /** null はルート直下。TRASH_ID はゴミ箱の中 */
    parent_id: NodeId | null,
    /** 同一 parent_id 内での並び順 */
    order: number,
    name: string,
};

type FolderNode = BaseNode & { type: 'folder', is_expanded: boolean };
type SheetNode  = BaseNode & { type: 'sheet' };
type Node = FolderNode | SheetNode;
```

シート本体はツリーに含めない。サイドバーの描画に必要なのは名前と階層だけであり、
本体まで抱えると起動時に全シートを読む羽目になる。

### 2.2 ワークスペース

```ts
type Workspace = {
    schema_version: 1,
    nodes: Record<NodeId, Node>,
    /** シート単位ではなくワークスペース単位で保持する(3.3 参照) */
    approved_domains: Domain[],
    active_sheet_id: NodeId | null,
};
```

### 2.3 シート本体

```ts
type Sheet = {
    id: NodeId,
    row_datas: RowData[],
    updated_at: number,
};
```

`RowData` は現行定義のまま変更しない。

`name` はツリー側(`Node.name`)にのみ持たせ、`Sheet` には重複させない。二箇所に
持つとリネーム時の同期漏れが必ず起きる。

## 3. 現行モデルからの変更点

### 3.1 `SaveData` の解体

`SaveData` は `Workspace` と `Sheet` に分割され、型としては廃止する。
マイグレーション用のレガシー型としてのみ残す。

| 現行フィールド | 移行先 |
|---|---|
| `project_name` | `Node.name` (シートノード) |
| `row_datas` | `Sheet.row_datas` |
| `approved_domains` | `Workspace.approved_domains` |

### 3.2 ストアの分割

現行の `useStore` は責務が混在しているため解体する。

| ストア | 責務 |
|---|---|
| `useWorkspaceStore` | ツリー、承認済みドメイン、行クリップボード、アクティブシートID |
| `useSheetStore` | アクティブシートの本体、選択状態、表示モード、Undo履歴 |
| `useModalStore` | 現行のまま |
| `useToastStore` | 現行のまま(4.3 でアクション付きトーストを追加) |

### 3.3 `approved_domains` の昇格

現行はシート相当のデータ(`SaveData`)の中にある。シート間で行をコピペする以上、
承認済みドメインがシートごとに違うと「コピペした行のURLが開けない」が発生する。
ワークスペース単位に上げる。

## 4. 機能仕様

### 4.1 サイドバー

- 画面左から**オーバーレイで**スライドインする。表の描画幅は変えない。既存の
  横スクロールや行ドラッグに影響させないため
- ヘッダとフッタ(どちらも `position: fixed`)を避けて配置する。高さは
  `main.css` の `--header-height` / `--footer-height` を参照する
- 開く入口は `HeaderControls` に置いた専用ボタン。ヘッダ左上のハンバーガーは
  工具箱全体のナビゲーションであり、別物
- ツリーはフォルダとシートを混在表示。フォルダは開閉可能(`is_expanded` を永続化)
- アクティブなシートを強調表示する
- 操作: 新規シート / 新規フォルダ / リネーム(ダブルクリック) / 削除(ゴミ箱へ) /
  ドラッグ&ドロップ移動

#### ドラッグ&ドロップ

ドロップゾーンは3種類。

| ゾーン | 効果 |
|---|---|
| ノードの上端 | 対象の直前に挿入(同じ親) |
| ノードの中央 | フォルダの子として挿入(フォルダのみ) |
| ノードの下端 | 対象の直後に挿入(同じ親) |

加えて、ツリーの空き領域はルート直下への移動、ゴミ箱の見出しはゴミ箱への移動
として扱う。

**挿入位置の計算**: `move_node` は移動対象を兄弟から除いてから挿入するため、
対象ノードの `order` をそのまま渡すと同じ親の中で1つずれる。`calc_drop_index`
で除外後の位置を求めてから渡すこと。

**循環チェック**: フォルダを自身の子孫へ移動する操作は拒否する。移動先候補の
`parent_id` を辿り、移動対象の `id` に到達したら不正とする。拒否された移動は
トーストを出さない(戻すものがないため)。

**既知の制約**: HTML5 の drag-and-drop を使っているため、タッチデバイスでは
動作しない。モバイル対応は別途必要。

### 4.1.1 シートが存在しない状態

シートが1枚もない状態は作らない。起動時・全削除後に該当した場合は空のシートを
自動生成する。UI の場合分けを減らすため。

### 4.2 シートの切り替え

- ツリー上のシートをクリックで切り替え
- 切り替え時: 現在のシートを保存 → 対象シートを IndexedDB からロード
- 未ロードのシートはメモリに載せない。イベント全体規模のシートが複数ある場合に
  全件を常駐させないため

### 4.3 削除とゴミ箱

Undo の対象がシート内編集に限られるため、ツリー操作は本質的に不可逆になる。
これを補うための仕組みを2段構えで用意する。

1. **ゴミ箱**: 削除は `parent_id = TRASH_ID` への移動として実装する。ゴミ箱は
   サイドバー最下部に常設し、明示的に空にするまでノードを保持する
2. **直前の移動を戻す**: D&D 実行時に移動前の `parent_id` / `order` を1手だけ
   保持し、トーストの「元に戻す」ボタンから復元できるようにする

トーストにアクションボタンを追加するため `useToastStore` を拡張する。

### 4.4 行の選択

現行の `selected_row_indexes` は2行上限がハードコードされており、かつ diff 表示の
トリガーを兼ねている。選択と比較対象指定を分離する。

```ts
/** 上限なし */
selection: number[],
/** Shift による範囲選択の起点 */
anchor_index: number | null,
display_mode: 'sum' | 'diff',
```

| 操作 | 挙動 |
|---|---|
| クリック | 単独選択。`anchor_index` を更新 |
| Ctrl + クリック | トグル。`anchor_index` を更新 |
| Shift + クリック | `anchor_index` から対象までを範囲選択 |

`display_mode` は明示的なトグルとする。`selection.length === 2` のときのみ diff を
選択可能にし、自動遷移はしない。複数行コピーのために2行選択した瞬間に表示が
切り替わるのを避けるため。

### 4.5 行のコピー&ペースト(シート間)

アプリ内クリップボードとして実装する。OS クリップボードは経由しない。

```ts
// useWorkspaceStore
row_clipboard: RowData[],
```

ワークスペースストアに置くことで、シートを切り替えても保持される。

| 操作 | 挙動 |
|---|---|
| Ctrl + C | `selection` の行を `row_clipboard` にコピー |
| Ctrl + X | 同上。コピー元の行を `INITIAL_ROW_DATA` にリセット |
| Ctrl + V | 選択行を起点に上書き |
| Ctrl + Shift + V | 選択行の直後に挿入 |

シートの行数を超える貼り付けは、超過分だけ行を自動追加する。

**既存の paste ハンドラとの衝突**: 現行の `App.vue` は取り込み用セルの `@paste` で
シミュ結果テキストを解析している。この動作は取り込み専用セルに限定されており、
通常セルでの Ctrl+V とはターゲットが異なるため共存できる。実装時に、取り込み用
セルにフォーカスがある場合はグローバルな Ctrl+V ハンドラを発火させないこと。

### 4.6 Undo / Redo

- スコープはアクティブシート内の `row_datas` 編集のみ
- ツリー操作(作成/削除/移動/リネーム)は対象外。4.3 のゴミ箱と移動取り消しで補う
- 履歴の上限は 50 件
- シート切り替え時に履歴をリセットする
- 履歴は永続化しない(メモリのみ)
- キーバインド: Ctrl+Z / Ctrl+Y

現行の `data_history` / `UNDO_DATA` / `REDO_DATA` は push 箇所が存在せず、どこからも
呼ばれていない死にコードになっている。この実装で置き換える。

### 4.7 共有URL

**発行**: 共有対象はアクティブなシート1枚。シート名・行データ・承認済みドメインを
旧 `SaveData` 形式に詰めて圧縮する。形式を変えていないため、既存の共有URLは
そのまま読める。

**取り込み**: 既存のシートを上書きせず、**ルート直下に新しいシートとして追加**して
アクティブにする。承認済みドメインは既存のものへマージする。

旧実装は「現在のデータを共有URLのデータによって上書きします」という confirm を
出していたが、これはこの仕様書が解消しようとしているデータ喪失そのものだった。
非破壊にしたことで confirm 自体が不要になった。

## 5. 永続化

### 5.1 IndexedDB

localStorage の 5MB 制限では、イベント全体規模のシートを複数保持する想定に耐えない。
IndexedDB に移行する。

| ストア | キー | 内容 |
|---|---|---|
| `workspace` | 固定キー `'current'` | `Workspace` 1件 |
| `sheets` | `NodeId` | `Sheet` |

DB 名 `cost-manager`、バージョン 1。

### 5.2 保存タイミング

- ワークスペース: ツリー変更のたびに保存
- シート本体: 編集のデバウンス保存(500ms) + シート切り替え時 + `beforeunload`

### 5.3 スキーマのバージョニング

`schema_version` を必須フィールドとする。現行の `SaveDataSchema` にはバージョンが
存在しないため、今後フィールドを追加すると旧データが `ValiError` になる。

さらに現行実装は、検証に失敗したデータを `localStorage.removeItem` で**黙って
削除している**。これを改める。

- 検証失敗時は削除せず、`cost-manager:corrupted:<timestamp>` として退避する
- ユーザーにモーダルで通知し、退避データのダウンロード手段を提供する

### 5.4 マイグレーション

`localStorage['cost-manager']` に旧 `SaveData` が存在する場合:

1. 旧データを `SaveDataSchema` で検証
2. ルート直下にシートノードを1件作成。`name` は `project_name`(空なら「無題のシート」)
3. `row_datas` を `Sheet` として `sheets` ストアへ
4. `approved_domains` を `Workspace.approved_domains` へ
5. **旧キーは削除せず** `cost-manager:backup` にリネームして残す

既存ユーザーのデータを壊した時点でツールの信用が失われるため、この経路には
必ずテストを書く(`tests/migration.test.ts`)。

  エクスポートとセットで別途検討する

- サーバーサイド同期・アカウント機能
- ツリー操作の Undo
